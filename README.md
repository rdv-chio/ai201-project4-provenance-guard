# Provenance Guard — Production Content Attribution Safety System

Provenance Guard is a backend API designed for creative sharing platforms to determine whether submitted content was authored by a human or generated by an Artificial Intelligence model. Instead of relying on rigid binary classifications, the system leverages a multi-signal detection pipeline, scores confidence dynamically to handle uncertainty honestly, enforces rate limiting to block automated abuse, maintains a structured SQLite audit log, and provides creators a transparent path to appeal automated decisions.

---

## Architecture Overview

A text submission traverses the backend pipeline through the following sequence:
1. **Ingress (`POST /submit`)**: The Flask routing layer intercepts the JSON payload, extracting the raw text and `creator_id`. It assigns a immutable `content_id` via UUIDv4 and sets an ISO 8601 timestamp.
2. **Multi-Signal Extraction**: 
   * **Signal 1 (Semantic/LLM)**: The text is dispatched to Groq's API (`llama-3.3-70b-versatile`) with structural constraints forcing a JSON response representing the semantic AI probability ($S_{llm}$).
   * **Signal 2 (Structural/Stylometric)**: Concurrently, a local Python engine tokenizes the text to compute **Sentence Length Variance (SLV)** and **Type-Token Ratio (TTR)** to output a structural AI score ($S_{heur}$).
3. **Calibrated Fusion**: The system evaluates both inputs using an asymmetric weighting formula to compute a combined confidence metric ($Score$).
4. **Transparency Routing**: The final score is mapped against calibrated boundaries to resolve a plain-language user status label.
5. **Persistence & Egress**: A structured log is committed to a local SQLite database (`provenance_guard.db`), and a `200 OK` JSON response is returned to the client.

## Detection Signals
To achieve high reliability, the system evaluates content across two completely independent linguistic dimensions:
1. Semantic & Stylistic Profiler (Groq API — `llama-3.3-70b-versatile`)
* **What it measures**: Holistic tone, choice of transitional words (e.g., "Furthermore," "Moreover," "It is important to note"), semantic consistency, and lack of stylistic risk-taking.
* **Why it differs**: Large Language Models optimize for maximum token probability sequence generation. This statistical smoothing results in highly standardized, predictable sentence transitions and cliché phrasing over extended passages.
* **Blind spot**: Highly formal, precise, or objective human writing (such as an academic abstract or a legal brief), which naturally communicates with uniform structures and structured progressions.

2. Stylometric Heuristic Engine (Pure Local Python)
* **What it measures**: Sentence Length Variance (SLV) and Type-Token Ratio (TTR) (vocabulary diversity metric).
* **Why it differs**: Natural human prose is structurally erratic. Authors intentionally weave short, punchy sentence clauses together with long, meandering run-ons, while scattering specialized vocabulary dynamically. AI text clusters tightly around tightly uniform sentence structures and safe, middle-tier vocabulary arrays to remain mathematically safe.
* **Blind spot**: Constrained human creative writing configurations, such as rhythmic poetry, fixed-meter epics, or minimalist flash fiction, which purposefully contract variations.

## Confidence Scoring & Calibration
Individual signal metrics are fused into a finalized calibration profile using an asymmetric formula:
$$Score = (0.65 \times S_{llm}) + (0.35 \times S_{heur})$$

### Rationale for Asymmetric Calibration
Falsely accusing an original human creator of passing off AI text as their own is a catastrophic platform failure mode that breaks creator trust. To safeguard human attribution, the formula weights the semantic model at 65% because it captures fluid contextual meaning, while treating the structural heuristic as a 35% balancing modifier.
The aggregate score defines three distinct user-facing categories:
* $0.00 \le Score < 0.40$: **Likely Human** — Strong variation and low semantic cliché density.
* $0.40 \le Score \le 0.75$: **Uncertain / Mixed Signals** — Minor conflicts between semantic flow and structural rhythm. The system explicitly defaults to a cautious buffer here rather than forcing a binary classification.
* $0.75 < Score \le 1.00$: **Likely AI-Generated** — Extreme uniformity coupled with distinct generative word markers.

## Transparency Label Design
The system rejects internal jargon like "logit scores" or "classifier weights" in favor of clear, plain-language text that provides context for non-technical users.

1. High-Confidence Human ($Score < 0.40$)

**Verified Human Attribution** — This content aligns consistently with human writing patterns and structural variance.

2. Uncertain Buffer ($0.40 \le Score \le 0.75$)

**Attribution Unverifiable** — This text contains a mixture of stylistic markers. Content context cannot be definitively automated.

3. High-Confidence AI ($Score > 0.75$)

**Automated Content Label** — Our systems indicate a high probability that this text was generated using an AI model.

## Production Infrastructure

### Rate Limiting Configuration
The `POST /submit` endpoint is governed by an explicit rate limit enforced via `Flask-Limiter` utilizing a fast, local memory storage layer:
* **Enforced Threshold**: `10 requests per minute` per unique IP address.
* **Operational Rationale**: Creative writing platforms cater to human storytellers who draft, copy, or submit finished chapters incrementally. A threshold of 10 submissions per minute comfortably accommodates a human user copying over multiple segments, while instantly blocking malicious scripts trying to scrape or flood the backend pipeline with synthetic materials.

### Structured SQLite Audit Log
Every pipeline event commits a comprehensive row entry into `provenance_guard.db`. This provides structural tracing showing exactly how a classification score was resolved:
* **Fields Logged**: `content_id` (UUIDv4), `creator_id`, `timestamp` (UTC ISO), `text_preview` (first 50 characters), `attribution`, `confidence` (combined score), `llm_score`, `heur_score`, `status` (`classified` or `under_review`), and `appeal_reasoning`.
* **API Access Hook**: The `GET /log` endpoint serializes these database rows directly to JSON format, ensuring absolute audit visibility.

### Appeals Workflow
When a creator disputes a classification, they issue a payload containing their qualitative explanation to `POST /appeal`.
1. The app runs a targeted lookup query verifying the submission's historical existence.
2. It transitions the lifecycle state column from `classified` to `under_review`.
3. It appends the raw text string containing the author's input straight into the `appeal_reasoning` column, making it instantly visible to a human moderator reviewing the audit log queue.

## Known Limitations
* **Specific Content Vulnerability (Linguistic Interference / ESL Authors)**:
Authors composing work in English as a second language (ESL) often rely on highly rigid, grammatically textbook sentence structures to avoid errors. This natural coping mechanism reduces global Sentence Length Variance and compresses vocabulary choices. As a result, the stylometric engine flags the text as uniform, which can push an otherwise benign human draft into the Uncertain category. The system mitigates this issue by providing the user an immediate path to appeal, acknowledging that automated systems have structural blind spots.

## Specification & Implementation Reflection
* **Plan Utility**: Defining the asymmetric signal combination formula and explicit boundary ranges in `planning.md` prior to code execution eliminated runtime ambiguity. It prevented the AI tools from defaulting to a lazy, standard binary split at `0.50`, ensuring the wide, human-protective "Uncertain" buffer zone was maintained perfectly.
* **Architectural Divergence**: The final implementation split the audit log out from raw text files into a structured local SQLite database (provenance_guard.db). A structured relational database schema proved necessary to cleanly modify and update the workflow status column during an appeal transaction without rewriting a static file on disk.

## AI Usage Log
* **Instance 1 (Scaffolding Connection Framework)**: Directed the AI to construct the core Flask routing stubs alongside a structured Groq interface client utilizing `response_format={"type": "json_object"}`. Override actions were taken to insert defensive `try/except` fallbacks that intercept network drops and default to a safe `0.5` uncertainty value rather than throwing unhandled server errors.
* **Instance 2 (Shell Syntax Diagnostics)**: Utilized AI to debug a persistent bash parser failure (`Expecting value: line 1 column 1 (char 0)`) occurring during multi-line string curls. The AI successfully diagnosed that internal single quotes in text contractions (like "I've") broke bash parameters. It was resolved by migrating to flattened single-line structures with escaped inner double quotes.
